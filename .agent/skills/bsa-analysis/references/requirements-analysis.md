# Requirements Analysis

## Elicitation Techniques

**Interviews**: 1:1 sessions with stakeholders; prepare questions, document responses
**Workshops**: Group sessions for consensus; use facilitator, timeboxed agenda
**Observation**: Watch users perform tasks; identify undocumented workflows
**Document Analysis**: Review existing specs, SOPs, system docs
**Prototyping**: Mockups/wireframes to validate understanding (see `prototyping.md`)
**Surveys**: Scale to many stakeholders; quantitative + qualitative questions

## Validation Checklist

- [ ] **Clear**: Unambiguous, single interpretation
- [ ] **Complete**: All info present, no TBD
- [ ] **Consistent**: No conflicts with other reqs
- [ ] **Testable**: Can write pass/fail test case
- [ ] **Traceable**: Linked to source and downstream
- [ ] **Feasible**: Technically achievable within constraints
- [ ] **Prioritized**: MoSCoW or priority assigned

## Common Pitfalls

- **Gold plating**: Adding unrequested features
- **Scope creep**: Uncontrolled requirement additions
- **Ambiguity**: "Fast", "user-friendly" without metrics
- **Missing NFRs**: Forgetting performance, security, scalability
- **No prioritization**: Everything is "high priority"
