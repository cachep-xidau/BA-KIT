# Prototyping

Validate requirements understanding through visual representations before development.

## Fidelity Levels

| Level | Purpose | Tools | When to Use |
|-------|---------|-------|-------------|
| Low | Quick concept validation | Paper, Balsamiq, Excalidraw | Early discovery, brainstorming |
| Mid | Layout & flow validation | Figma, Sketch, draw.io | Requirements refinement |
| High | Visual design & interactions | Figma, Adobe XD, Framer | Stakeholder sign-off, dev handoff |

## Prototype Types

**Wireframe**: Structural layout, no styling - focus on information hierarchy
**Mockup**: Static visual design - colors, typography, imagery
**Interactive**: Clickable prototype - user flows, transitions
**Paper**: Hand-drawn sketches - fastest iteration, disposable

## Prototyping Process

```
Identify Goals → Select Fidelity → Create → Review → Iterate → Document
     ↓              ↓              ↓        ↓         ↓          ↓
  What to       Low/Mid/High    Build    Stakeholder Refine   Specs/
  validate                      screens   feedback   based on  handoff
                                                    feedback
```

## Best Practices

- **Start low**: Paper/low-fi first, increase fidelity as requirements solidify
- **Focus on flows**: Validate user journeys before visual polish
- **Disposable mindset**: Don't over-invest in early prototypes
- **Document assumptions**: Note what's placeholder vs confirmed
- **Test early**: Get feedback before adding detail

## Common Mistakes

- Starting with high-fidelity too early
- Treating prototype as final design
- Not documenting feedback systematically
- Skipping stakeholder validation
- Over-engineering throwaway prototypes
